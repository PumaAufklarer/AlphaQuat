## Architecture: feature engineering (`src/alpha_quat/features/`)

| Module | Purpose |
|--------|---------|
| `factor.py` | `Factor` dataclass (name, expression, category, depends_on). `compile()` translates DSL to DuckDB SQL via regex |
| `registry.py` | `FactorRegistry` ‚Äî topo sort (Kahn's BFS), cycle detection, `min_lookback()` |
| `engine.py` | `FeatureEngine` ‚Äî builds 2-CTE DuckDB query (ts + rank), computes per batch of dates |
| `pipeline.py` | `FeaturePipeline` ‚Äî date scheduling with incremental/rebuild/--since, batch of 20 dates |
| `alphasets/` | Pre-built factor sets. `alpha158.py` ‚Üí `build_alpha158()` returns a `FactorRegistry` |

**Computation model:**

```
raw (base CTE: read daily/*.parquet with vwap column)
  ‚îÇ
  ‚ñº
_ts CTE (all time-series + RANK/QUANTILE inner exprs, WINDOW w_time)
  ‚îÇ
  ‚ñº
_rank CTE (all RANK()/NTILE() OVER expressions, no WINDOW)
  ‚îÇ
  ‚ñº
SELECT ts_code, trade_date, <all factors> WHERE trade_date BETWEEN min AND max
```

Only **2 CTEs** regardless of factor count ‚Äî never create one CTE per factor.

## Gotchas

- **Stale `__pycache__`** ‚Äî if behavior seems wrong after a code change, `find . -name __pycache__ -not -path './.venv/*' -exec rm -rf {} +`
- **`start_date` on DataSource** ‚Äî for APIs with limited history (e.g. `stock_st` only has data from 2016), set `start_date = "20160101"` on the subclass to skip useless queries
- **Future dates excluded** ‚Äî Pipeline auto-filters trade_cal dates > today
- **Tushare rate limits** ‚Äî minute-level, Fetcher defaults to 12 retries at 5s intervals (60s total)
- **`stock_st` is the API name** ‚Äî tushare pro uses `stock_st` (not `stk_st`)

### Feature engineering gotchas

- **`$vwap` is a column, not inline SQL** ‚Äî vwap is computed in the engine's base CTE (`amount / NULLIF(vol, 0) AS vwap`). `compile()` replaces `$vwap` ‚Üí `vwap` (plain column ref). Never expand `$vwap` to inline SQL ‚Äî it breaks downstream `\w+` regex patterns.
- **RANK/QUANTILE must be two CTEs** ‚Äî wrapping a time-series expression inside RANK (e.g. `RANK(REF(...))`) requires an inner CTE (with `w_time` window) and an outer CTE (pure ranking, no `w_time`). DuckDB rejects nested window functions. The engine handles this automatically ‚Äî do not attempt single-CTE for RANK/QUANTILE factors.
- **Batch processing** ‚Äî pipeline processes dates in batches of 20. Each batch builds the CTE chain once and filters by date range (`WHERE trade_date BETWEEN`). Do not change to per-date processing without re-benchmarking (single date: 0.86s, batch of 20: ~3s).
- **Factor dependencies** ‚Äî alpha158 factors only depend on `$raw` fields, not other factors. The topo sort exists for future factor sets that build chains of derived factors. Adding a cycle between factors raises `ValueError`.

## Architecture: strategy (`src/alpha_quat/strategy/`)

| Module | Purpose |
|--------|---------|
| `types.py` | `StrategyContext` (trade_date, capital, universe, prices, prev_holdings, constraints), `SignalResult`, `StrategyResult` ‚Äî all dataclasses |
| `signal.py` | `ISignalGenerator` ABC ‚Äî one abstract method: `generate(features, ctx) -> SignalResult` |
| `position.py` | `IPositionManager` ABC ‚Äî three abstract methods: `allocate(signals, ctx) -> DataFrame`, `constrain(positions, ctx) -> DataFrame`, `execute(target, prev, ctx) -> tuple[DataFrame, DataFrame]` |
| `strategy.py` | `Strategy` ‚Äî concrete pipeline orchestrator, constructor-injects `ISignalGenerator` + `IPositionManager`, `run()` chains: generate ‚Üí allocate ‚Üí constrain ‚Üí execute |

**Pipeline contract:**

```
feature DataFrame (ts_code, trade_date, factors...)
    ‚îÇ
    ‚ñº
ISignalGenerator.generate(features, ctx)          ‚Üí SignalResult (ts_code, score)
    ‚îÇ
    ‚ñº
IPositionManager.allocate(signals, ctx)            ‚Üí positions (ts_code, target_weight)
    ‚îÇ
    ‚ñº
IPositionManager.constrain(positions, ctx)         ‚Üí positions (clamped weights)
    ‚îÇ
    ‚ñº
IPositionManager.execute(positions, prev, ctx)     ‚Üí (positions, orders)
    ‚îÇ
    ‚ñº
StrategyResult(target_positions, orders, metadata)
```

**Key rules:**

- **Strategy code never does I/O** ‚Äî all data flows in via method parameters (DataFrame, StrategyContext). No file reading, no API calls inside strategy code.
- **DataFrame schema is the contract** between stages ‚Äî no intermediate DTO classes.
- **Signal answers "what to trade"** (score). **Position answers "how much and how"** (weight, shares, orders).
- **Dependency injection** ‚Äî `Strategy(signal, position)`; any ISignalGenerator + any IPositionManager works.
- **Pipeline order is immutable** ‚Äî generate ‚Üí allocate ‚Üí constrain ‚Üí execute. No skipping stages.
- **`prev_holdings` can be `None`** ‚Äî first trading day, no prior positions. `execute()` must handle it.
- **`target_shares` is computed in `execute()`** ‚Äî needs price from `ctx.prices` (not in `allocate()`).

## Architecture: backtesting (`src/alpha_quat/backtest/`)

| Module | Purpose |
|--------|---------|
| `config.py` | `BacktestConfig` dataclass ‚Äî start/end dates, capital, monthly_addition, commission, stop_loss, top_k, factor names |
| `filters.py` | `build_universe(date, data_dir)` ‚Äî filters to main board (`stock_basic.market == "‰∏ªÊùø"`) excluding ST stocks |
| `portfolio.py` | `Portfolio` ‚Äî cash, holdings (100-share lots), snapshots, trade log. `buy()`/`sell()` with commission |
| `engine.py` | `BacktestEngine` ‚Äî day-by-day loop: monthly addition, stop loss, T+1 open execution, mark-to-market |
| `metrics.py` | `compute_metrics()` ‚Äî cumulative/annualized return, max drawdown, Sharpe, win rate |
| `report.py` | `generate_html_report()` ‚Äî self-contained HTML with matplotlib charts (base64 embedded) |

**Timing model:** Signal at close of day T (from `features/YYYYMMDD.parquet`) ‚Üí execute at open of day T+1. Stop loss checked at T-1 close ‚Üí sell at T open.

**Engine does NOT use `Strategy.run()`** ‚Äî calls `ISignalGenerator.generate()` at EOD and `IPositionManager` methods at next open, bypassing the synchronous `Strategy.run()` chain. This is required for T+1 execution timing.

**Existing strategy implementations:**
- `strategy/signals/ma_cross.py` ‚Äî `MACrossSignal`: golden/dead cross detection via `KLEN35`/`KLEN36` factors
- `strategy/positions/equal_weight.py` ‚Äî `EqualWeightTopKPosition`: top-K equal weight allocation with share lot rounding

### Backtest gotchas

- **features path uses YYYYMMDD (no dashes)** ‚Äî unlike daily/ which uses hive-style `YYYY_MM_DD`. `BacktestEngine` reads `features/{trade_date}.parquet` where trade_date is `20240115` not `2024_01_15`.
- **Must run fetch+feature before backtest** ‚Äî `engine.py` reads `data/daily/`, `data/features/`, `data/stock_basic.parquet`, `data/stock_st/`, and `data/trade_cal.parquet`. All must exist.
- **matplotlib is a required dep** ‚Äî added for HTML report charts. Listed in `pyproject.toml`.

## Adding a new data source

1. Create `src/alpha_quat/data/sources/<name>.py` as a DataSource subclass with api_name, partition_by, fields
2. Create `tests/test_sources/test_<name>.py` (test api_name, partition_by, get_params, path_for)
3. Register in `cli.py` `ALL_SOURCES` dict
4. If data doesn't go back to 1990, set `start_date` on the class

## Adding a new factor set

1. Create `src/alpha_quat/features/alphasets/<name>.py` with a `build_<name>() -> FactorRegistry` function
2. Add to `cli.py` `ALL_FEATURE_SETS` dict as `"name": "alpha_quat.features.alphasets.<name>:build_<name>"`
3. Define `Factor` instances using the DSL: `$close`, `$volume`, `$amount`, `$open`, `$high`, `$low`, `$vwap` for raw fields; `REF(f, N)`, `MEAN(f, N)`, `STD(f, N)`, `SUM(f, N)`, `MAX(f, N)`, `MIN(f, N)`, `CORR(f1, f2, N)`, `DELTA(f, N)`, `RANK(f)`, `QUANTILE(f, N)` for operators
4. Create `tests/test_features/test_<name>.py` verifying: all compile, no cycles, valid deps, lookback consistent
